# Agent Context Passing & Shared Memory — Design Research

Research into how agents share context and coordinate information across workflow steps. Covers the current state, explicit handoff notes, shared scratchpads, context windowing, and agent-to-agent tagging.

**Current state:** Agents share information through two mechanisms: (1) the `executionContext` JSON array on work items, which accumulates `ExecutionContextEntry` records (executionId, summary, outcome, rejectionPayload) as agents execute, and (2) the `project_memories` module (`memory.ts`) which generates compressed summaries when top-level work items reach Done. There are no explicit handoff notes, no scratchpad, no agent-to-agent tagging, and no context windowing. Each agent receives the full `executionContext` array and optionally project memories via the `get_context` MCP tool.

---

## 1. Current State — How Agents Share Information

### What exists today

**A. ExecutionContext on work items**

File: `packages/shared/src/entities.ts` (line 70), `packages/backend/src/db/schema.ts` (line 33)

Each work item has an `executionContext` JSON array that stores one entry per execution:

```typescript
interface ExecutionContextEntry {
  executionId: ExecutionId;
  summary: string;           // One-line summary from the completing agent
  outcome: ExecutionOutcome; // "success" | "failure" | "rejected" | ...
  rejectionPayload: RejectionPayload | null;
}
```

This is injected into the next agent's system prompt by `buildSystemPrompt()` in `claude-executor.ts` (line 78):

```
## Previous Executions
- [success] Implemented login form with email/password fields
- [rejected] Code review found missing input validation
  Rejection: Missing XSS sanitization on email field (severity: major)
  Hint: Use DOMPurify before rendering user input
```

**Limitations:**
- **Summary only.** The summary is a single line generated by the completing agent. No structured data about files changed, decisions made, or open questions.
- **No persona attribution.** The entry doesn't record *which persona* produced the result — just the outcome and summary.
- **No filtering.** Every agent in every state sees the entire history, regardless of relevance.
- **Linear accumulation.** Entries are appended but never pruned. After many workflow cycles (e.g., 10+ engineer→reviewer→engineer loops), the history becomes long and noisy.

**B. Project memories**

File: `packages/backend/src/agent/memory.ts`

When a top-level work item reaches Done, the memory module generates a structured summary via a Haiku one-shot call:

```typescript
interface ProjectMemory {
  summary: string;        // 1-3 sentence summary
  filesChanged: string[]; // File paths mentioned
  keyDecisions: string[]; // Important choices made
}
```

Memories are available to agents via the `get_context` MCP tool (with `includeMemory: true`). They're project-scoped and have a consolidation mechanism that merges the oldest half when count exceeds 50.

**Limitations:**
- **Only at Done.** Memories are generated when work completes — not during intermediate workflow steps. An agent in "In Review" can't see memories from the current in-progress work.
- **Project-scoped, not item-scoped.** Memories are about the project as a whole. There's no way to get memories specifically about the current work item's lineage.
- **Read-only during execution.** An agent can read memories but can't write to them mid-execution. The memory is only generated post-completion.
- **No inter-item context.** If work item A discovers something relevant to work item B, there's no mechanism to share that knowledge except through project-level memories (which are generated only after A is Done).

### What's lost between workflow steps

When work item "Fix auth bug" transitions from In Progress (Engineer) → In Review (Code Reviewer):

| Information | Available to Code Reviewer? | Source |
|---|---|---|
| One-line summary of what Engineer did | Yes | `executionContext[n].summary` |
| Outcome (success/fail) | Yes | `executionContext[n].outcome` |
| Rejection feedback (if previously rejected) | Yes | `executionContext[n].rejectionPayload` |
| Which files the Engineer modified | **No** | Lost (not in ExecutionContextEntry) |
| Why the Engineer made specific design choices | **No** | Lost (was in agent's thinking) |
| What the Engineer tried and abandoned | **No** | Lost |
| Open questions the Engineer had | **No** | Lost |
| Warnings about fragile areas of the codebase | **No** | Lost |
| Intermediate notes during long executions | **No** | Not possible to write |

## 2. Explicit Handoff Notes

### Concept

When an agent finishes a workflow step, it writes a structured handoff document summarizing its work for the next agent. This is richer than the current one-line summary and follows a defined schema.

### Handoff note structure

```typescript
interface HandoffNote {
  id: string;
  executionId: ExecutionId;
  workItemId: WorkItemId;
  fromPersonaId: PersonaId;
  fromPersonaName: string;
  fromState: string;           // Workflow state this was produced in
  targetState?: string;        // Intended next state (if known)
  targetPersonaHint?: string;  // "@reviewer" tag (see §5)

  // Structured content
  summary: string;             // 2-5 sentence summary of work done
  filesChanged: FileChange[];  // Files modified with change type
  decisionsMade: Decision[];   // Key choices with rationale
  openQuestions: string[];      // Unresolved issues for the next agent
  warnings: string[];           // "Be careful with X" notes
  context: Record<string, unknown>; // Arbitrary structured data

  createdAt: Date;
}

interface FileChange {
  path: string;
  changeType: "added" | "modified" | "deleted";
  description?: string;        // What was changed and why
}

interface Decision {
  decision: string;            // What was decided
  rationale: string;           // Why
  alternatives?: string[];     // What was considered and rejected
}
```

### How the agent writes it

The handoff note is generated at the end of execution, before the agent's final response. Two approaches:

**Option A: Agent writes it explicitly via MCP tool**

Add a `write_handoff` MCP tool that the agent calls as its last action:

```
write_handoff({
  summary: "Implemented login form with email/password fields...",
  filesChanged: [{ path: "src/auth/login.tsx", changeType: "added", description: "New login component" }],
  decisionsNote: [{ decision: "Used react-hook-form", rationale: "Better validation than manual state" }],
  openQuestions: ["Should we add OAuth support?"],
  warnings: ["The auth middleware is tightly coupled to Express - changing it affects all routes"]
})
```

**Option B: System generates it from execution events**

After execution completes, a post-processing step (similar to the current memory generation) analyzes the execution's tool calls and text output to extract:
- Files changed: from `Write`, `Edit`, `Bash` tool calls
- Decisions: from the agent's reasoning text (harder to extract reliably)
- Summary: from the agent's final message

**Recommendation: Hybrid.** Use Option B for files-changed (reliable — just parse tool calls) and Option A for decisions/questions/warnings (agent knows its reasoning better than a post-processor can infer).

### How the next agent receives it

The handoff note replaces the flat `executionContext` summary in the system prompt:

```
## Handoff from Engineer (In Progress → In Review)

### Summary
Implemented login form with email/password fields. Used react-hook-form for
validation. Added bcrypt password hashing in the API route.

### Files Changed
- `src/auth/login.tsx` (added) — New login component with form validation
- `src/api/auth.ts` (modified) — Added POST /auth/login endpoint
- `src/middleware/session.ts` (modified) — Session cookie configuration

### Decisions
- Used react-hook-form over manual state management — better validation DX
- bcrypt with cost factor 12 — balances security and performance for local app

### Open Questions
- Should we add OAuth (Google/GitHub) support in this iteration?

### Warnings
- The session middleware is tightly coupled to Express req/res — will need
  refactoring if we switch to Hono or Fastify middleware syntax
```

This is substantially more useful to the Code Reviewer than the current:
```
## Previous Executions
- [success] Implemented login form with email/password fields
```

### Storage

**Option 1: New `handoff_notes` table**

```sql
CREATE TABLE handoff_notes (
  id TEXT PRIMARY KEY,
  execution_id TEXT NOT NULL REFERENCES executions(id) ON DELETE CASCADE,
  work_item_id TEXT NOT NULL REFERENCES work_items(id),
  from_persona_id TEXT NOT NULL REFERENCES personas(id),
  from_state TEXT NOT NULL,
  summary TEXT NOT NULL,
  files_changed TEXT NOT NULL DEFAULT '[]',   -- JSON: FileChange[]
  decisions_made TEXT NOT NULL DEFAULT '[]',  -- JSON: Decision[]
  open_questions TEXT NOT NULL DEFAULT '[]',  -- JSON: string[]
  warnings TEXT NOT NULL DEFAULT '[]',        -- JSON: string[]
  context TEXT NOT NULL DEFAULT '{}',         -- JSON: arbitrary
  created_at INTEGER NOT NULL
);
```

**Option 2: Extend `executionContext` JSON on work items**

Replace the current `ExecutionContextEntry` with a richer structure that includes handoff data. Simpler (no new table) but loses the benefits of queryable columns.

**Recommendation: Option 1** — a dedicated table is cleaner, queryable, and can be joined to executions for audit/analytics. The `executionContext` column on work items becomes a lightweight pointer array (execution IDs) rather than carrying all the data.

## 3. Shared Scratchpad

### Concept

A per-work-item scratch space where agents can read and write notes during execution — not just at completion. Distinct from comments (which are for human-facing communication) and handoff notes (which are structured, end-of-execution summaries).

### Use cases

| Scenario | How scratchpad helps |
|---|---|
| Engineer finds a bug while implementing a feature | Writes a note: "Found race condition in UserService.save() — not related to current task but @reviewer should check" |
| Planning agent decomposes a story | Writes architecture sketch and constraints that child task agents need to follow |
| Long-running execution | Agent writes intermediate checkpoints: "Step 1/3 done: API routes created. Step 2: frontend components" |
| Cross-step data passing | Engineer writes structured data (API schema, DB migration notes) that the Reviewer and QA agents need |

### Design

```typescript
interface ScratchpadEntry {
  id: string;
  workItemId: WorkItemId;
  executionId?: ExecutionId;    // Which execution wrote this (null for manual entries)
  personaId?: PersonaId;        // Which persona (null for human entries)
  tag?: string;                 // Optional category: "architecture", "todo", "warning", "data"
  targetPersona?: string;       // "@reviewer" — see §5
  content: string;              // Freeform text or structured data
  pinned: boolean;              // Pinned entries persist through context windowing
  createdAt: Date;
}
```

### How it differs from existing concepts

| Concept | Purpose | Written by | When | Visible to |
|---|---|---|---|---|
| **Comments** | Human-facing discussion | Humans, system events | Anytime | UI, activity feed |
| **Handoff notes** | Structured end-of-step summary | Agent (at execution end) | After execution | Next agent in workflow |
| **Scratchpad** | Intermediate notes during execution | Agent (during execution), human | Anytime | Other agents on same work item |
| **Project memories** | Project-level learnings | System (post-completion) | After work item Done | All agents in project |

### MCP tools for scratchpad

```
write_scratchpad({
  workItemId: "wi-abc",
  tag: "architecture",
  content: "Using repository pattern for DB access — all queries go through repo classes",
  pinned: true
})

read_scratchpad({
  workItemId: "wi-abc",
  tag?: "architecture",          // Optional filter
  targetPersona?: "reviewer",   // Optional filter
  limit?: 20
})
```

### Storage

```sql
CREATE TABLE scratchpad_entries (
  id TEXT PRIMARY KEY,
  work_item_id TEXT NOT NULL REFERENCES work_items(id) ON DELETE CASCADE,
  execution_id TEXT REFERENCES executions(id),
  persona_id TEXT REFERENCES personas(id),
  tag TEXT,
  target_persona TEXT,
  content TEXT NOT NULL,
  pinned BOOLEAN NOT NULL DEFAULT false,
  created_at INTEGER NOT NULL
);
```

### Frontend visibility

Scratchpad entries can appear in the work item detail panel as a collapsible "Agent Notes" section — separate from the comment stream. This gives humans visibility into what agents are thinking without cluttering the conversation view.

## 4. Context Windowing

### Problem

As a work item passes through many workflow steps, accumulated context grows:
- 5+ handoff notes from different personas
- 20+ scratchpad entries
- Project memories
- Work item description, parent chain, inherited context

At some point, injecting all of this into the system prompt exceeds what's useful (or what fits in the context window).

### Strategy: Tiered relevance with budget allocation

Allocate a token budget to context injection (e.g., 4,000 tokens) and fill it by priority:

```
Priority 1 (always included):
  - Work item title, description, current state
  - Parent chain (if any)
  - Most recent handoff note (from the immediately preceding step)

Priority 2 (included if budget allows):
  - Pinned scratchpad entries (marked important by agents)
  - Open questions from all handoff notes
  - Warnings from all handoff notes

Priority 3 (included if budget allows):
  - Older handoff notes (summarized, not full text)
  - Recent unpinned scratchpad entries
  - Project memories (via getRecentMemories with remaining budget)

Priority 4 (omitted, available via MCP tool):
  - Old scratchpad entries
  - Full text of older handoff notes
  - Consolidated project memories
```

### Implementation approach

```typescript
interface ContextWindow {
  budget: number;               // Total tokens allocated
  sections: ContextSection[];   // Priority-ordered sections
}

interface ContextSection {
  priority: 1 | 2 | 3 | 4;
  label: string;                // "Recent Handoff", "Pinned Notes", etc.
  content: string;              // Pre-rendered text
  tokenEstimate: number;        // ~content.length / 4
  required: boolean;            // Priority 1 sections are always included
}

function buildContextWindow(
  workItemId: string,
  budget: number = 4000,
): ContextWindow {
  // 1. Collect all context sources
  // 2. Estimate tokens for each
  // 3. Fill by priority until budget exhausted
  // 4. For excluded content: make available via get_context MCP tool
}
```

### Summarization of older handoff notes

When handoff notes are pushed out of the active context window, summarize them into a compressed form:

```
## Earlier Steps (summarized)
- Planning (Planner): Decomposed into 3 subtasks. Key decision: use REST over GraphQL.
- Implementation (Engineer): Built API routes and React components. 8 files changed.
- First review (Reviewer): Rejected — missing error handling in 2 API routes.
- Rework (Engineer): Added try/catch blocks and error responses.
```

This summarization can use the same Haiku summarizer pattern from `memory.ts` — a cheap one-shot call that compresses N handoff notes into a paragraph.

### Configurable budget

The context window budget should be configurable at:
1. **Project level** — `projects.settings.contextBudget` (default: 4000 tokens)
2. **Persona level** — some personas need more context (planners) or less (quick lint checks)

### On-demand retrieval via MCP

Content that doesn't fit in the system prompt is still accessible. The `get_context` MCP tool already exists — extend it to support:

```
get_context({
  workItemId: "wi-abc",
  includeMemory: true,
  includeHandoffs: true,      // NEW: return all handoff notes
  includeScratchpad: true,    // NEW: return scratchpad entries
  handoffLimit: 5,            // NEW: limit for pagination
})
```

This way, agents that need deep historical context can request it, while agents doing simple tasks don't pay the context cost.

## 5. Agent-to-Agent Tagging

### Concept

An agent in one workflow step can flag information for a specific persona in a later step. For example: "@reviewer: pay attention to the auth changes in routes/login.ts" or "@qa: test the edge case where email is empty".

### Syntax

Tags use the `@persona_name` convention in both handoff notes and scratchpad entries:

```typescript
// In handoff note
{
  targetPersonaHint: "reviewer",
  warnings: ["@reviewer: The auth middleware changes affect all protected routes — test /api/user and /api/admin specifically"]
}

// In scratchpad
{
  targetPersona: "reviewer",
  content: "Pay attention to the error handling in auth.ts — I'm not sure the 401 vs 403 distinction is correct"
}
```

### How tagged content is surfaced

When building the context window for a specific persona, tagged content gets elevated priority:

```typescript
function getRelevanceBoost(entry: ScratchpadEntry | HandoffNote, currentPersona: Persona): number {
  // Exact persona name match
  if (entry.targetPersona?.toLowerCase() === currentPersona.name.toLowerCase()) {
    return 2;  // Double priority — always include
  }
  // Role-based match (e.g., "@reviewer" matches persona with name "Code Reviewer")
  if (entry.targetPersona && currentPersona.name.toLowerCase().includes(entry.targetPersona.toLowerCase())) {
    return 1.5;
  }
  return 1;
}
```

Tagged content aimed at the current persona is promoted to Priority 1 in the context window — it's always included, even if the budget is tight.

### Tag resolution

Tags are soft — they're hints, not hard references. Resolution:

1. **Exact match:** `@engineer` matches persona named "Engineer"
2. **Substring match:** `@reviewer` matches "Code Reviewer"
3. **No match:** Tag is preserved but doesn't get priority boost. The content is still visible to all agents.

No validation at write time — agents can tag persona names that don't exist yet (the persona might be added later in the workflow).

### UI integration

In the work item detail panel, tagged entries could be highlighted or filtered:
- "Show notes for: [All] [Engineer] [Code Reviewer] [QA]"
- Tagged notes show a colored badge: "@reviewer" in blue

## 6. Implementation Phases

### Phase 1: Handoff notes (highest value, moderate effort)

- Add `handoff_notes` table
- Add `write_handoff` MCP tool for agents
- Auto-extract `filesChanged` from tool call events post-execution
- Modify `buildSystemPrompt()` to include the most recent handoff note instead of flat `executionContext` summaries
- Keep `executionContext` array for backward compatibility but deprecate it as the primary context source

**Why first:** Handoff notes solve the biggest gap — agents currently get almost no useful context about what the previous agent did. This single change dramatically improves multi-step workflow quality.

### Phase 2: Context windowing (prevents prompt bloat)

- Implement `buildContextWindow()` with tiered priority
- Add token budget to project and persona settings
- Summarize older handoff notes using Haiku
- Extend `get_context` MCP tool for on-demand deep retrieval

**Why second:** As soon as handoff notes exist, context size grows. Windowing prevents the system from degrading as work items pass through many steps.

### Phase 3: Scratchpad (enables mid-execution communication)

- Add `scratchpad_entries` table
- Add `write_scratchpad` / `read_scratchpad` MCP tools
- Add "Agent Notes" section to work item detail panel in the UI
- Support for pinned entries

**Why third:** Scratchpad is useful but less critical than handoff notes. Handoff notes cover 80% of the context-sharing need. Scratchpad adds value for complex, multi-agent scenarios.

### Phase 4: Agent tagging (refinement)

- Add `targetPersona` field to handoff notes and scratchpad entries
- Implement relevance boost in context window builder
- Add tag-based filtering in the UI
- Consider: tag autocomplete from available persona names in the MCP tool

**Why last:** Tagging is a refinement on top of handoff notes and scratchpad. It makes context sharing more targeted but isn't strictly necessary — agents can read all notes regardless.

## 8. Design Decisions

### Handoff notes are separate from comments

Comments are human-facing discussion. Handoff notes are agent-to-agent structured data. Mixing them would clutter the comment stream with machine-generated content, and would make it harder to query/filter agent context separately.

### Scratchpad is per-work-item, not per-execution

An execution-scoped scratchpad would be lost when the execution ends. A work-item-scoped scratchpad persists across executions, enabling cross-step communication. Entries still reference the execution that created them for traceability.

### Context windowing uses token estimation, not exact counting

Exact token counting requires a tokenizer (tiktoken or similar). For context window budgeting, character-based estimation (~4 chars/token) is sufficient — this is the same approach used in `memory.ts` (line 23: `APPROX_CHARS_PER_TOKEN = 4`). The budget is a soft limit, not a hard cutoff.

### Tags are soft references, not foreign keys

Hard-referencing persona IDs would require the tagged persona to exist at write time. Soft name-based tags are more flexible: an agent can tag "@qa" even if no QA persona is configured yet. Resolution happens at read time.

### executionContext backward compatibility

The existing `executionContext` array on work items continues to work as-is. Handoff notes are additive — they provide richer context alongside the existing summaries. Migration path: once handoff notes are established, `buildSystemPrompt()` can prefer handoff notes and only fall back to `executionContext` entries for executions that predate the feature.

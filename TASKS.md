## Research: Webhooks & External Triggers

> Design research for inbound triggers from external systems and outbound event webhooks. Output is design docs only — do NOT add implementation tasks to TASKS.md. All proposals go to `docs/proposals/webhooks/`.

- [ ] **RES.WEBHOOKS.INBOUND** — Research inbound triggers from external systems. Investigate: (1) GitHub integration — trigger a workflow when a PR is opened, issue is created, CI fails, or review is requested; webhook receiver endpoint + GitHub App or webhook secret verification, (2) generic webhook receiver — `POST /api/webhooks/:triggerId` that maps incoming payloads to workflow execution; user-configurable payload-to-prompt mapping (extract PR title, diff URL, issue body into the agent's prompt), (3) Slack triggers — slash command or bot mention that kicks off an agent task ("@agentops review this PR"); app setup flow, (4) trigger configuration UX — in Settings or per-project; define trigger name, source (GitHub/Slack/generic), event filter, target workflow + persona, prompt template with payload variables, (5) security — webhook secret validation (HMAC), rate limiting, IP allowlisting, replay protection; how to handle malformed or malicious payloads. Write to `docs/proposals/webhooks/inbound-triggers.md`. Commit the doc only.

- [ ] **RES.WEBHOOKS.OUTBOUND** — Research outbound event webhooks. Investigate: (1) what events can be subscribed to — execution started/completed/failed, workflow step changed, budget threshold reached, work item status changed, (2) webhook configuration — URL, secret for HMAC signing, event type filter, per-project or global scope; test/ping button to verify endpoint, (3) payload format — standardized JSON envelope with event type, timestamp, and event-specific data; include enough context to be useful without leaking sensitive info (API keys, full prompts), (4) delivery — async dispatch with retry (exponential backoff, max 5 retries), delivery log with status codes and response bodies visible in Settings, dead letter handling for persistently failing endpoints, (5) relationship to notifications (RES.NOTIFY.*) — are outbound webhooks a notification channel (alongside Slack/email), or a separate developer-facing system; can notifications be built on top of the webhook infrastructure. Write to `docs/proposals/webhooks/outbound-events.md`. Commit the doc only.
